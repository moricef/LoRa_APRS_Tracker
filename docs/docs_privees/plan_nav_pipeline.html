Refactorisation pipeline NAV — tuile par tuile

LoRa APRS Tracker — branche feature/refactor-map-engine — mars 2026

Le probleme

Le pipeline actuel charge toutes les tuiles (7-9) en PSRAM avant de dispatcher et rendre les features. A Z9, chaque tuile fait ~440 KB. Le budget PSRAM est epuise avant meme de commencer le rendu.

Pipeline actuel (batch)

1. Charger 7 tuiles → 3060 KB
2. Dispatcher features → +300 KB
3. Rendre geometrie → +200 KB
4. Rendre labels
Total: ~3500 KB — PSRAM dispo: ~3100 KB

Pipeline cible (streaming)

1. Charger tuile N → 440 KB
2. Rendre geometrie → +50 KB
3. Collecter labels → +2 KB
4. Liberer tuile N, tuile N+1...
5. Rendre tous les labels
Pic: ~500 KB — largement dans le budget

Architecture cible

renderNavViewportStreaming() — Z9-Z11

fillSprite(bgColor)
pour chaque tuile (ordre spirale) :
charger tuile (cache hit ou SD read)
setClipRect(tileOffsetX, tileOffsetY, 256, 256)
pour chaque feature (ordre du fichier = z-order intra-tuile) :

renderNavViewport() — Z12+ (inchange)

Pipeline batch actuel conserve : tuiles petites (~20 KB), z-ordering cross-tuiles correct, PSRAM suffisante. Aucun changement necessaire.

Z-ordering

Compromis accepte : le pipeline streaming perd le z-ordering cross-tuiles. Un polygone prio 5 de la tuile A sera rendu avant un polygone prio 0 de la tuile B.

A Z9 (50 km/tuile), l'overlap aux frontieres est minime. IceNav fait exactement pareil. L'artefact visuel est negligeable.

Le z-ordering intra-tuile est preserve : les features sont dans l'ordre du fichier .nav (genere par le tile generator avec le bon z-order).

Etapes d'implementation

  1. Extraire le rendu d'une feature
    Fonction renderFeature(map, fp, ref, ...) : le switch cases 1/2/3 actuel, deplace dans une fonction autonome. Utilise par les deux pipelines.
  2. Creer renderNavViewportStreaming()
    Nouveau pipeline tuile-par-tuile. Charge une tuile, itere ses features, rend immediatement la geometrie, collecte les labels.
  3. Dispatcher selon le zoom
    Dans renderNavViewport() : Z9-Z11 appelle streaming, Z12+ garde le batch. Seuil configurable.
  4. Gestion cache adaptative
    A Z9-Z11 : limiter navCache a 3-4 tuiles max. Liberer agressivement apres rendu. A Z12+ : garder le NAV_CACHE_SIZE=12 actuel.
  5. Test + validation device
    Verifier : toutes les tuiles rendent a Z9, PSRAM stable, pas d'artefact aux frontieres, labels corrects, performance acceptable (<3s).

Risques et mitigations

RisqueImpactMitigation
Artefacts aux frontieres de tuiles moyen setClipRect par tuile empeche le debordement. A verifier visuellement.
Labels coupes aux bords de tuiles faible Les labels sont dans textRefs avec clearClipRect — rendus apres toute la geometrie, pas de clipping.
Performance (SD reads sequentiels) moyen Le navCache amortit les relectures. Ordre spirale = centre d'abord = visible en premier.
Regression Z12+ faible Pipeline batch inchange pour Z12+. Pas de risque de regression.

Pansement en place (en attendant)

Les tuiles centrales (ordre spirale) sont chargees en priorite. Les coins sont sacrifies si PSRAM insuffisante.
